# codespud — agent guide

> Entry point for AI agents working in this folder. Read this first.
> Type: code (Gatsby static site) + prose (markdown blog content)
> Last updated: 2026-06-20

## Purpose

Source for **codespud.com**, Chris Bautista's personal coding blog. It is a Gatsby v2
static site that builds a set of markdown posts (under `content/`) into a deployed
website. Two kinds of content live here: dated **blog** posts and a **works**
portfolio. A small custom Node toolchain (`scripts/`) handles the author workflow:
scaffold a draft, list drafts, publish, and generate a status dashboard.

## Scope & boundaries

- In scope: writing/editing blog posts and works entries (`content/`), the React +
  styled-components templates and components that render them (`src/`), Gatsby config
  (`gatsby-*.js`), and the publishing scripts (`scripts/`).
- Out of scope: anything that isn't the codespud.com site.
- Safe to change: `content/blog/<year>/*.md`, `content/works/*.md`, `src/**`,
  `gatsby-config.js`, `gatsby-node.js`, `scripts/**`.
- Do not touch: `public/`, `.cache/`, `node_modules/` (all generated), and the legacy
  WordPress import under `content/blog/wp-content/`. See Guardrails.

## Guardrails

Hard rules and when to stop and ask a human.

- **Never hand-edit `public/`** — it is the generated build output (`gatsby build`).
  Edits there are overwritten on the next build and never appear in source. Change
  `src/` or `content/` instead, then rebuild.
- **Never edit `.cache/` or `node_modules/`** — generated/vendored.
- **Leave `content/blog/wp-content/` alone** — legacy assets from the old WordPress
  site, kept only so historical post URLs/images still resolve.
- **Never remove a `redirect_from:` entry or change a published post's `url:`** without
  adding a matching redirect — these preserve inbound links from the WordPress era and
  from social shares. See ADR-0003.
- **Never commit secrets.** Analytics/AdSense IDs in `gatsby-config.js` and `Notes.md`
  are public client IDs and are fine; anything that looks like a private key/token is not.
- Stop and ask a human before: deleting posts, bulk-renaming slugs, deploying
  (`npm run deploy`/`upload` pushes live to production), or upgrading Gatsby's major
  version (this is pinned to v2; see Gotchas).

## Conventions

**Run / build (commands run from the folder root):**

- `npm install` — install deps (Node + npm; Gatsby v2).
- `npm run develop` (alias `start`) — local dev server at `http://localhost:8000`;
  GraphQL explorer at `http://localhost:8000/___graphql`.
- `npm run build` — production build into `public/`.
- `npm run serve` — serve the built `public/` locally.
- `npm run clean` — clear `.cache/` and `public/` (do this when the build behaves oddly).
- `npm run format` — Prettier over `src/**/*.{js,jsx}` (config in `.prettierrc`).
- `npm run deploy` — push `public/` to the `gh-pages` branch (production). `npm run upload`
  = `build` then `deploy`. **Production action — confirm with a human first.**

**Post format (markdown + YAML frontmatter).** Blog posts live at
`content/blog/<year>/<slug>.md`. Frontmatter fields, as produced by `new-post.js`:

```yaml
---
title: Human Readable Title
author: chris
type: post
date: 2026-05-14 13:49:00+00:00      # "YYYY-MM-DD HH:MM:SS+00:00"
url: /2026/aria-live-regions-deep-dive/   # /<year>/<slug>/
redirect_from:
  - /aria-live-regions-deep-dive/     # bare-slug redirect, preserves old URLs
featured_image: ""                    # path under content/blog/assets or /blog/...
featured_image_attribution: ""
tags: [accessibility, ai, web development]
draft: true                           # true = excluded from the build
---
```

- Posts with `draft: true` are **filtered out of the build** (see `gatsby-node.js`).
  Publishing means flipping it to `false` — use `npm run publish`, don't hand-edit.
- **Works** entries (`content/works/*.md`) use a different frontmatter shape: `path`,
  `categories`, `excerpt`, and crucially `contentType: works`. `gatsby-node.js` splits
  blog vs works on `contentType`.
- Body uses Markdown; `<!--more-->` marks the excerpt cutoff. Code blocks get Prism
  highlighting; CodePen embeds via `@weknow/gatsby-remark-codepen`.
- Tags are kebab-cased into tag pages at `/tags/<tag>/`. There is a permanent redirect
  `/tags/console` → `/tags/cli` in `gatsby-node.js`.

## Common tasks

- **Write a new post:** `npm run new-post -- "My Post Title"` → creates
  `content/blog/<currentYear>/<slug>.md` as a draft with frontmatter pre-filled. Write
  the body, fill `tags`, and set `featured_image` (+ attribution).
- **See what's unpublished:** `npm run drafts` — lists every `draft: true` post with its
  age and whether tags/image are missing.
- **Publish a draft:** `npm run publish -- content/blog/2026/my-post.md` — flips
  `draft: true` → `false` and refreshes `date` to now. It warns (does not block) if tags
  or featured_image are empty. Then `npm run develop` to preview, then deploy.
- **Refresh the status dashboard:** `npm run dashboard` — regenerates `dashboard.html`
  from current content (post counts, drafts, tags, etc.).
- **Edit how posts render:** change templates in `src/templates/` (e.g.
  `blog-post.js`) or shared `src/components/`. Restart `npm run develop` after changing
  `gatsby-node.js` or `gatsby-config.js`.
- **Deploy to production:** `npm run upload` (build + push to `gh-pages`). Confirm first.

## How to verify your work

There is no unit-test suite (`npm test` is a placeholder). Verification is build- and
review-based:

- **Build passes:** `npm run clean && npm run develop` (or `npm run build`) completes
  with no GraphQL/markdown errors. A broken frontmatter field or bad image path will fail
  the build here — this is the main safety net.
- **Post renders:** visit the post's `url:` on `localhost:8000`; confirm title, body,
  code highlighting, featured image, and tag links all resolve.
- **Content checklist (for a new/edited post):** `draft` is set correctly, `tags`
  non-empty, `featured_image` set, `url` matches `/<year>/<slug>/`, `redirect_from` kept.
  `npm run drafts` surfaces missing tags/images.
- **Formatting:** `npm run format` leaves `src/` clean.

## Reference example

- A good **published blog post** looks like: `content/blog/2025/ai-accessibility-audits.md`
  (complete frontmatter: tags, featured_image + attribution, `draft: false`).
- A good **works entry** looks like: `content/works/boxerjs.md`.

## Glossary

- **works** — the portfolio section (projects), as opposed to dated blog posts.
  Distinguished by `contentType: works` in frontmatter.
- **draft** — `draft: true` frontmatter; excluded from the production build until published.
- **redirect_from** — frontmatter list of old paths that should 301 to this post; how
  legacy WordPress and short URLs keep working (via `gatsby-plugin-meta-redirect`).
- **wp-content** — leftover asset tree from the site's WordPress era.
- **codespud** — the brand/site name; the "potato" icon (`content/assets/potato.png`) and
  `@codespud` handle come from it.

## Gotchas

- **Gatsby is pinned to v2** (`gatsby: ^2.3.16`, React 16). Don't casually upgrade —
  many plugins here are v2-era and a major bump is a migration project, not a chore.
- **`package-lock.jsonxxx`** sits next to `package-lock.json` and looks real. It isn't —
  it's a stray backup. Use `package-lock.json` only.
- **`public/` is committed** but generated. Don't hand-edit it and don't treat its
  contents as source; the source is `content/` + `src/`.
- **`draft: true` posts silently vanish** from the build (filtered in `gatsby-node.js`).
  If a new post "isn't showing up", check `draft`.
- **Two date formats exist:** blog posts use `YYYY-MM-DD HH:MM:SS+00:00`; older works
  entries use ISO `...T..Z`. Match the format already in the file you're editing.
- Restart the dev server after editing `gatsby-node.js` / `gatsby-config.js` — those run
  at bootstrap, not on hot-reload.
